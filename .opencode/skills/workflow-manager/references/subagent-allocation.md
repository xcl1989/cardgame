# 子代理任务分配指南

关于何时以及如何启动子代理进行并行任务执行的详细指导。

---

## 决策框架

### 第 1 步：任务分析

收到复杂请求时，使用此清单分析：

**需要问的问题：**
1. 这个任务能否拆分为独立的子任务？
2. 并行执行这些子任务能否显著节省时间？
3. 是否有子任务需要在隔离环境中更好地使用工具（搜索、探索）？
4. 是否有 3+ 个不同的研究或探索活动？

**任一为 YES：** 考虑启动子代理。

---

### 第 2 步：并行化评估

使用此评分系统：

| 因素 | 1 分（不并行） | 3 分（考虑） | 5 分（肯定并行） |
|------|---------------|-------------|-----------------|
| **独立性** | 所有任务互相依赖 | 部分任务独立 | 大部分任务完全独立 |
| **数量** | 1-2 个子任务 | 3-4 个子任务 | 5+ 个子任务 |
| **复杂度** | 都简单（各<2 分钟） | 混合复杂度 | 都复杂（各>10 分钟） |
| **工具类型** | 都需要主上下文 | 部分可用子代理 | 多个需要专门代理 |
| **总时间** | <10 分钟 | 10-30 分钟 | >30 分钟 |

**行动阈值：**
- **5-10 分**：在主上下文保持
- **11-18 分**：考虑为 1-2 个最大任务启动子代理
- **19-25 分**：并行启动多个子代理

---

### 第 3 步：子代理类型选择

#### 以下情况使用 `general` 代理：

- **研究任务**："查找关于 X、Y、Z 的信息"
- **多步骤分析**："分析这个数据并生成报告"
- **代码编写**："写一个实现 X 的函数"
- **复杂推理**："比较这些方案并推荐一个"
- **内容创建**："为 X 写文档"

**general 代理提示结构：**
```
[背景]：简要解释整体目标
[任务]：清楚说明要完成什么
[要求]：列出具体要求
[输出格式]：描述返回格式
[验证]：提及任何需要的验证
```

#### 以下情况使用 `explore` 代理：

- **文件搜索**："查找匹配模式 X 的所有文件"
- **代码库探索**："功能 X 在哪里实现？"
- **模式匹配**："查找函数 Y 的所有使用"
- **关键词搜索**："搜索 API Z 的引用"
- **结构分析**："项目架构是什么？"

**深度级别：**

| 级别 | 何时使用 | 示例 |
|------|---------|------|
| `quick` | 单一模式，特定位置 | "在 src/models/ 中查找 User 类" |
| `medium` | 多个模式，中等范围 | "查找所有 React 组件及其 props" |
| `very thorough` | 全面搜索，多个位置 | "在整个代码库中查找所有 API 端点" |

**explore 代理提示结构：**
```
[范围]：定义搜索内容
[模式]：列出要查找的模式
[位置]：指定目录或排除某些路径
[输出]：描述期望的返回格式
```

## 常见模式

### 模式 1：多源研究

**场景**："研究 X、Y、Z 的最佳方案"

**子代理分配：**
```
子代理 1 (general): 研究 X - 描述、优缺点、示例
子代理 2 (general): 研究 Y - 描述、优缺点、示例  
子代理 3 (general): 研究 Z - 描述、优缺点、示例
```

**为何并行**：每个研究线索独立；结果稍后合并。

---

### 模式 2：代码库探索

**场景**："查找认证是如何实现的以及工作原理"

**子代理分配：**
```
子代理 1 (explore, very thorough): 查找所有认证相关文件
子代理 2 (explore, quick): 查找认证配置文件
子代理 3 (explore, medium): 查找代码中认证使用模式
```

**为何并行**：不同搜索模式可以并发运行。

---

### 模式 3：功能实现

**场景**："添加一个带数据可视化的新仪表盘"

**子代理分配：**
```
子代理 1 (general): 研究此用例的最佳图表库
子代理 2 (explore, medium): 查找现有仪表盘组件以遵循模式
子代理 3 (explore, quick): 查找已在使用中的数据获取工具
[主上下文]：使用收集的信息实现功能
```

**为何并行**：研究和探索可以在主上下文准备实现时进行。

---

### 模式 4：数据分析

**场景**："分析我们的销售、市场和 product 数据以发现趋势"

**子代理分配：**
```
子代理 1 (general): 分析销售数据 - 加载、清洗、计算指标、发现趋势
子代理 2 (general): 分析市场数据 - 加载、清洗、计算指标、发现趋势
子代理 3 (general): 分析产品数据 - 加载、清洗、计算指标、发现趋势
[主上下文]：综合发现，创建合并报告
```

**为何并行**：每个数据集分析独立；综合在之后进行。

---

### 模式 5：文档生成

**场景**："为我们的整个后端创建 API 文档"

**子代理分配：**
```
子代理 1 (explore, very thorough): 查找所有 API 端点定义
子代理 2 (explore, medium): 查找请求/响应 schema
子代理 3 (general): 研究 API 文档最佳实践
[主上下文]：使用收集的信息编写文档
```

**为何并行**：探索和研究可以并发进行。

## 启动模板

启动多个子代理时，使用此格式：

```
<launch_parallel_subagents>

**子代理 1：[名称/角色]**
- **type**: general | explore
- **description**: [2-5 词描述]
- **prompt**: [完整指令，包含背景、任务、要求、输出格式]
- **thoroughness**: quick | medium | very thorough（仅 explore）

**子代理 2：[名称/角色]**
- **type**: general | explore
- **description**: [2-5 词描述]
- **prompt**: [完整指令]

[更多子代理...]

</launch_parallel_subagents>
```

## 最佳实践

### 应该做：

✅ **为独立任务启动子代理**
- 并行文件搜索
- 多个研究线索
- 对不同项目的批量操作

✅ **编写自包含的提示**
- 包含所有必要的上下文
- 指定确切的输出格式
- 提及验证步骤

✅ **批量启动**
- 在一条消息中启动所有独立子代理
- 不要在启动下一个独立代理前等待一个完成

✅ **审查和综合**
- 等待所有子代理返回
- 合并和交叉引用结果
- 如有差距，通过后续提问填补

### 不应该做：

❌ **不要并行化依赖任务**
- 如果 B 需要 A 的输出，保持顺序
- 在 A 完成前不要启动 B

❌ **不要过度使用子代理**
- 简单任务（<2 分钟）不需要子代理
- 不要启动 10+ 子代理；逻辑分组

❌ **不要忘记整合**
- 子代理结果对用户不可见
- 你必须综合并呈现发现
- 不要只说"子代理发现 X"；解释这意味着什么

❌ **不要无明确目的启动**
- 每个子代理应该有具体、定义明确的任务
- 不要说"探索代码库"；说"查找所有 React 组件"

## 错误处理

### 如果子代理失败或返回不完整信息：

1. **评估影响**：没有这个信息能否继续？
2. **重试**：用更清晰的提示再次启动子代理
3. **回退**：使用替代方法（直接搜索、问用户）
4. **继续**：不要让一个失败的子代理阻塞所有进度

### 如果子代理结果冲突：

1. **向用户确认冲突**
2. **调查**：启动后续子代理
3. **呈现两个版本**，让用户决定
4. **在最终输出中注意不确定性**

## 示例

### 好的子代理启动：

```
**子代理 1：API 研究**
- type: general
- description: 研究 Stripe 支付 API
- prompt: 研究 Stripe 支付 API 处理信用卡扣款。返回：(1) 认证方法，(2) 创建扣款的端点，(3) 必需参数，(4) 常见错误码及处理。包含代码示例。对照官方 Stripe 文档验证信息。

**子代理 2：查找现有代码**
- type: explore
- description: 搜索支付代码
- thoroughness: medium
- prompt: 在代码库中搜索现有支付集成代码。查找：(1) Stripe SDK 或类似支付库的导入，(2) 名为"charge"、"payment"、"checkout"的函数，(3) 支付事件的 webhook 处理器。返回每个匹配的文件路径和函数名。
```

### 坏的子代理启动：

```
**子代理 1：**
- type: explore
- description: 四处看看
- prompt: 探索代码库看看能找到什么。

// 太模糊，没有具体任务，输出不明确
```
