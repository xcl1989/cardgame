# 工作流程模板

常见工作流程模式的详细示例。

---

## 模板 1：Bug 修复工作流程

### 触发
用户报告代码中的 bug 或错误。

### 步骤

1. **重现问题**
   - 请求错误消息、堆栈跟踪或重现步骤
   - 如可能，尝试在本地重现
   - 确认理解预期行为与实际行为

2. **定位来源**
   - 使用 `grep` 搜索错误消息或相关代码
   - 使用 `glob` 查找相关文件
   - 阅读可疑代码部分

3. **诊断根本原因**
   - 追踪执行流程
   - 识别具体的行或逻辑错误
   - 考虑可能触发 bug 的边界情况

4. **规划修复**
   - 为修复步骤创建待办
   - 考虑修复是否会影响其他区域
   - 选择最小、最安全的修复

5. **实施修复**
   - 进行代码变更
   - 如修复不明显，添加注释
   - 如存在测试，更新测试

6. **验证修复**
   - 运行测试
   - 手动验证 bug 已解决
   - 检查是否有回归

7. **沟通**
   - 解释 bug 的原因
   - 描述应用的修复
   - 如适当，建议预防措施

## 模板 2：功能实现工作流程

### 触发
用户请求新功能或功能。

### 步骤

1. **澄清需求**
   - 询问功能应该做什么
   - 识别边界情况和约束
   - 确认验收标准

2. **探索现有模式**
   - 查找代码库中类似功能
   - 理解架构和约定
   - 识别新代码应该放在哪里

3. **设计实现**
   - 拆分为子任务
   - 识别依赖
   - 规划实现顺序

4. **增量实现**
   - 从核心逻辑开始
   - 添加集成点
   - 最后构建 UI/接口（如适用）

5. **持续测试**
   - 继续前验证每个子任务
   - 运行现有测试防止回归
   - 如项目有测试，添加新测试

6. **最终审查**
   - 验证所有需求已满足
   - 检查代码风格与现有代码匹配
   - 确保没有调试代码或注释残留

7. **文档**
   - 为复杂逻辑添加内联注释
   - 如需要新设置，更新 README
   - 向用户解释用法

## 模板 3：代码审查工作流程

### 触发
用户要求审查代码、检查问题或提高代码质量。

### 步骤

1. **理解范围**
   - 审查哪些文件或部分？
   - 关注哪些方面？（bug、风格、性能、安全）
   - 有任何具体担忧吗？

2. **系统审查**
   - 逻辑性地阅读代码
   - 检查常见问题：
     - 错误处理
     - 边界情况
     - 资源泄漏
     - 安全漏洞
     - 性能瓶颈

3. **检查约定**
   - 命名一致性
   - 代码风格对齐
   - 文档完整性

4. **提供反馈**
   - 按严重程度列出问题（严重、中等、轻微）
   - 建议具体修复
   - 解释原因

5. **提供修复**
   - 询问用户是否希望你实施修复
   - 优先处理严重问题

## 模板 4：数据分析工作流程

### 触发
用户有数据需要分析、洞察或报告。

### 步骤

1. **理解目标**
   - 分析应该回答什么问题？
   - 这将为哪些决策提供信息？
   - 结果应该采取什么格式？

2. **探索数据**
   - 加载并检查数据结构
   - 检查数据类型、缺失值、异常值
   - 理解列之间的关系

3. **清洗和准备**
   - 适当处理缺失值
   - 根据需要转换数据
   - 如有用，创建派生列

4. **分析**
   - 计算相关统计
   - 寻找模式和趋势
   - 如适用，测试假设

5. **可视化**
   - 创建回答核心问题的图表
   - 使用适当的图表类型
   - 确保清晰可读

6. **解释**
   - 解释数据显示什么
   - 强调关键洞察
   - 注意任何限制或警告

7. **交付**
   - 提供发现摘要
   - 包含可视化
   - 建议下一步或后续问题

## 模板 5：学习/探索工作流程

### 触发
用户想要理解代码库、库或技术。

### 步骤

1. **识别学习目标**
   - 用户想理解什么？
   - 他们当前的知识水平如何？
   - 实际应用还是概念理解？

2. **收集资源**
   - 查找相关文档
   - 定位示例代码
   - 识别要涵盖的关键概念

3. **结构化探索**
   - 从高级概述开始
   - 深入具体组件
   - 展示具体示例

4. **检查理解**
   - 询问解释是否清楚
   - 提出深入特定主题
   - 提供关键点摘要

5. **提供参考**
   - 链接到官方文档
   - 建议练习
   - 注意常见陷阱

## 模板 6：多代理并行研究工作流

### 触发
复杂任务包含多个独立的研究或探索线索。

### 步骤

1. **分析任务结构**
   - 列出所有子任务
   - 识别哪些是独立的
   - 估算每个的时间
   - 使用并行化决策矩阵评分

2. **设计子代理分配**（如评分≥19）
   - 为每个独立线索分配子代理
   - 选择正确的类型（general/explore）
   - 为 explore 设置深度级别
   - 编写具体的提示

3. **并行启动子代理**
   - 在单条消息中启动所有独立子代理
   - 使用 `<launch_parallel_subagents>` 格式
   - 为每个提供清晰、自包含的提示

4. **等待全部完成**
   - 审查每个子代理的输出
   - 检查完整性和准确性
   - 如有需要，启动后续子代理填补缺口

5. **综合结果**
   - 合并所有子代理的发现
   - 解决任何冲突
   - 交叉引用相关信息

6. **继续主任务**
   - 使用收集的信息规划实现
   - 创建待办列表
   - 执行并验证

### 示例子代理启动

```
<launch_parallel_subagents>

**子代理 1：UI 组件研究**
- type: general
- description: 研究个人资料 UI 模式
- prompt: 研究 React 中用户个人资料页的最佳实践。查找：(1) 个人信息显示，(2) 可编辑表单字段，(3) 订单历史表格。返回推荐的组件结构及 shadcn/ui 或类似库的代码示例。

**子代理 2：用户 API 探索**
- type: explore
- description: 查找用户 API 代码
- thoroughness: medium
- prompt: 在代码库中搜索用户相关的 API 端点。查找：(1) 包含"user"和"api"或"route"的文件，(2) getUser、updateUser 函数，(3) 用户 schema 或模型定义。返回文件路径和端点 URL。

**子代理 3：订单 API 探索**
- type: explore
- description: 查找订单 API 代码
- thoroughness: medium
- prompt: 在代码库中搜索订单相关的 API 端点。查找：(1) 包含"order"和"api"或"route"的文件，(2) getOrders、getOrderHistory 函数，(3) 订单 schema 或模型定义。返回文件路径和端点 URL。

**子代理 4：认证模式发现**
- type: explore
- description: 查找认证代码
- thoroughness: quick
- prompt: 查找此代码库中如何处理认证。查找：(1) 认证中间件或守卫，(2) 会话/令牌管理代码，(3) 受保护路由示例。返回文件路径和关键函数名。

</launch_parallel_subagents>
```

### 时间节省

**示例计算：**
- 4 个独立研究任务，每个约 8 分钟
- 顺序：4 × 8 = 32 分钟
- 并行：Max(8, 8, 8, 8) = 8 分钟
- **节省：24 分钟（75% 更快）**

## 模板选择指南

| 场景 | 使用模板 |
|------|---------|
| 用户报告错误 | Bug 修复 |
| 用户请求新功能 | 功能实现 |
| 用户要求检查代码 | 代码审查 |
| 用户有数据需要分析 | 数据分析 |
| 用户想学习某事 | 学习/探索 |
| 复杂多线索任务 | 多代理并行研究 |
| 不确定 | 从功能实现开始，按需调整 |
