# 工作流程管理器

本技能提供管理复杂多步骤任务并确保一致执行的结构化方法。

## 核心工作流程

### 1. 任务分解工作流程

收到复杂任务时，遵循以下流程：

**第 1 步：分析请求**
- 识别主要目标
- 列出已知约束和需求
- 如有歧义，向用户提问（使用 `question` 工具）

**第 2 步：分解为子任务**
- 拆分为逻辑有序的步骤
- 每个子任务应该是原子且可验证的
- 估算复杂度（快速 < 5 分钟，中等 < 30 分钟，长时间 > 30 分钟）

**第 3 步：识别并行化机会**
- 寻找可以并发执行的独立子任务
- 使用下方的**并行化决策矩阵**判断何时使用 `task` 工具
- 将相关子任务分组给同一个子代理

**第 4 步：创建执行计划**
- 使用 `todowrite` 工具创建结构化待办列表
- 按依赖关系排序任务
- 标记高优先级项目
- 标注哪些任务将使用子代理

**第 5 步：执行并检查**
- 为并行任务启动子代理（见下方子代理工作流程）
- 逐个完成顺序子任务
- 验证输出后再继续
- 每步完成后更新待办状态

---

### 2. 并行化决策矩阵

**使用 `task` 工具启动子代理的时机：**

✅ **应该启动子代理：**
- 多个独立的文件搜索（如"查找所有 React 组件和所有 API 端点"）
- 并行研究任务（如"查询 X、Y、Z 三个库的文档"）
- 并发文件读取（如"读取 5 个不同的配置文件"）
- 独立分析（如"分析销售数据、市场数据和产品数据"）
- 批量操作（如"为 20 张图片生成缩略图"）

❌ **不应启动子代理：**
- 紧密依赖的任务（B 任务需要 A 任务的输出）
- 简单操作（< 2 分钟）
- 需要与用户持续交互的任务
- 需要查看之前对话上下文的操作

**子代理选择：**
- `general` - 用于复杂多步骤任务、研究、编写代码
- `explore` - 用于代码库探索、文件搜索、查找模式

**子代理深度（针对 explore 代理）：**
- `quick` - 基础搜索，单一模式
- `medium` - 多个模式，中等深度
- `very thorough` - 跨多个位置的全面分析

**参见：** [references/subagent-allocation.md](references/subagent-allocation.md) 获取详细决策框架和示例。

### 3. 子代理工作流程

**启动子代理时：**

1. **确定范围**
   - 子代理应该完成什么具体任务？
   - 应该返回什么信息给你？
   - 任务是否自包含？

2. **编写清晰的提示**
   - 包含具体的操作指令
   - 指定期望的返回格式
   - 提及验证步骤（如"测试你编写的代码"）
   - 设定期望深度

3. **尽可能并行启动**
   - 多个独立子代理可以在一条消息中启动
   - 批量启动相似的子代理

4. **等待完成**
   - 子代理返回单条结果消息
   - 检查输出是否完整
   - 如有需要请求后续跟进

5. **整合结果**
   - 合并多个子代理的输出
   - 验证结果的一致性
   - 使用收集的信息继续下一步

**子代理启动示例：**

```
<launch_parallel_subagents>

**子代理 1：API 文档研究**
- type: general
- description: 研究 Stripe 支付 API
- prompt: 研究 Stripe 支付 API 的支付端点。返回摘要：(1) 认证方法，(2) 创建扣款的必需参数，(3) 错误处理模式。包含代码示例。

**子代理 2：支付代码查找**
- type: explore
- description: 查找支付集成代码
- thoroughness: medium
- prompt: 在代码库中搜索现有的支付集成代码。查找：(1) Stripe SDK 的使用，(2) 支付处理函数，(3) webhook 处理器。返回文件路径和关键函数名。

</launch_parallel_subagents>
```

### 4. 进度追踪与邮件通知工作流程

**保持状态可见：**
- 实时更新 `todowrite` 列表
- 开始时标记为 `in_progress`，完成时标记为 `completed`
- 复杂任务添加简短完成说明

**里程碑邮件通知：**

当完成重要里程碑时，使用 `email-sender` skill 发送邮件通知用户。

**收件人：** `49781545@qq.com`

**触发邮件的场景（仅此 3 种）：**
- ✅ **任务结束** - 整个任务或主要阶段完成
- ✅ **遇到问题** - 技术难题、阻塞无法继续
- ✅ **需要确认** - 方向选择、方案决策需要用户拍板

**不要发送邮件的场景：**
- ❌ 仅完成几个子任务
- ❌ 定时进度报告
- ❌ 正常进行中

**邮件发送步骤（必须执行）：**

1. **生成邮件内容** - 使用下方模板填写实际内容
2. **调用 email-sender 发送** - 必须实际执行发送命令，不能只描述
   ```bash
   使用 email-sender skill:
   - 收件人：49781545@qq.com
   - 主题：[任务完成/遇到问题/需要确认] xxx
   - 内容：[邮件正文]
   ```
3. **确认发送成功** - 检查返回结果是否为 `"success": true`
4. **记录已发送** - 在待办中记录，避免重复发送

**重要：** 触发邮件条件时，必须实际调用 `email-sender` skill 发送邮件，不能仅用文字描述邮件内容。

**邮件内容模板：**
```
主题：[工作流进度] {任务名称} - {完成百分比}%

正文：
## 当前进度

✅ 已完成：
- {已完成的任务列表}

⏳ 进行中：
- {当前正在执行的任务}

⏸️ 待执行：
- {剩余任务列表}

## 执行统计
- 开始时间：{HH:MM}
- 已用时长：{X 小时 Y 分钟}
- 已完成任务数：{N}/{Total}

## 下一步计划
{接下来要执行的 1-3 个任务}

## 需要决策（如有）
{需要用户确认或决策的事项}
```

**示例邮件：**
```
主题：[工作流进度] 用户评价功能开发 - 60%

## 当前进度

✅ 已完成：
- 研究用户评价 UI 组件最佳实践
- 查找现有的 Product 组件和 API
- 查找用户认证和权限代码
- 查找数据库 schema 和模型定义

⏳ 进行中：
- 实现评价功能后端 API

⏸️ 待执行：
- 实现评价功能前端组件
- 添加表单验证
- 编写测试用例
- 集成测试

## 执行统计
- 开始时间：14:30
- 已用时长：0 小时 25 分钟
- 已完成任务数：4/9

## 下一步计划
1. 完成后端 API 实现
2. 开始前端组件开发
3. 添加输入验证

发送自：opencode 工作流程管理器
```

**处理阻塞：**
- 如被阻塞，标记任务状态为 `cancelled`
- 向用户解释阻塞原因
- 如阻塞超过 10 分钟，发送邮件通知
- 提出替代方案

**提供进度摘要：**
- 完成 3+ 任务后，总结已完成内容
- 切换上下文前，回顾当前状态
- 每小时自动发送进度邮件（长任务）

### 5. 质量检查点工作流程

对需要验证的任务，实施检查点：

**执行前检查：**
- 验证前置条件已满足
- 确认理解需求
- 检查潜在风险或边界情况

**执行中验证：**
- 关键操作后验证结果
- 运行测试或 lint（如适用，见验证命令）
- 及早发现错误，避免累积

**执行后审查：**
- 验证所有需求已满足
- 运行最终验证命令
- 确认无意外副作用

## 工具使用模式

### 有效使用 `todowrite`

```markdown
1. 在多步骤任务 START 时创建待办
2. 保持描述简洁但具体
3. 使用优先级：关键路径为 high，可选为 low
4. 每步完成后 IMMEDIATELY 更新状态
5. 如执行过程中范围扩大，添加新待办
```

### 使用 `question` 工具

以下情况提问：
- 存在多种有效方案
- 需求不明确
- 用户偏好影响实现
- 需要用户输入权衡

**好的提问格式：**
- 提供提问背景
- 提供 2-4 个具体选项及权衡
- 如有偏好，添加 "(Recommended)" 标签

### 使用 `bash` 进行验证

代码变更后始终运行验证命令：

```bash
# 检查项目是否有 lint/typecheck 命令
npm run lint    # 如存在 package.json
npm run typecheck
ruff check .    # 如为 Python 项目
python -m pytest  # 如存在测试
```

## 常见工作流程模板

### 软件开发工作流程
1. 理解需求 → 提问澄清
2. 探索代码库 → 查找相关文件
3. 规划变更 → 创建待办列表
4. 实施 → 一次一个子任务
5. 验证 → 运行测试/lint
6. 审查 → 验证所有需求
7. 提交（如请求）→ 遵循 git 协议

### 数据分析工作流程
1. 理解分析目标
2. 探索数据结构
3. 清洗/准备数据
4. 执行分析
5. 可视化结果
6. 记录发现
7. 验证结论

### 文档编写工作流程
1. 识别受众和目的
2. 收集源材料
3. 创建大纲
4. 起草内容
5. 审查清晰度
6. 添加示例/代码片段
7. 最终校对

## 处理中断

当用户在流程中改变方向时：

1. **暂停当前任务** - 标记为 `in_progress`（不要完成）
2. **确认转向** - 确认理解新请求
3. **保留上下文** - 保持待办可见以便恢复
4. **干净恢复** - 返回时，回顾进行中的内容

---

## 文件操作协议

**编辑文件前：**
- 先读取文件（永远不要假设内容）
- 检查现有模式/约定
- 识别最小改动

**进行变更时：**
- 优先编辑而非创建新文件
- 完全匹配现有代码风格
- 进行聚焦的原子变更

**编辑后：**
- 验证编辑成功
- 如适用，运行验证命令
- 更新待办状态

---

## 最佳实践

1. **一次一件事** - 顺序完成子任务，不要并行
2. **早验证，常验证** - 不要等到最后才验证
3. **沟通状态** - 复杂任务时让用户了解进度
4. **知道何时停止** - 不要过度设计；投入匹配任务范围
5. **提交前提问** - 未经明确请求永远不要 git 提交/推送
6. **保留用户意图** - 有疑问时，澄清而非假设
