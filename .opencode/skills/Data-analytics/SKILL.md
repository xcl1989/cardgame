# 数据分析助手

## 概述

基于大模型能力直接调用数据 API 进行数据模型查询和分析。

### 核心能力
- 获取所有数据模型的完整结构信息
- 根据用户问题智能推荐相关模型（1-5 个）
- 使用指标库改写用户问题
- 将复杂问题智能拆分为多个可执行的子问题
- 大模型自主规划并执行查询流程
- 使用计算器工具确保数值计算准确
- 智能汇总分析结果，生成可视化报告

## 步骤 2：选择模型并改写问题

**🚨 关键规则**：
- ✅ 字段名必须带表前缀（如 `collectionReg3X.receivingAmount`）
- ✅ 必须直接复制步骤 1 的完整 `columnName`
- ✅ 必须使用指标库改写用户问题
- ✅ 输出 JSON 后立即继续步骤 3，不要等待确认

**输出格式**:
```json
{
  "selectedModels": [
    {
      "tableName": "模型名称",
      "description": "模型描述",
      "columns": [
        ["字段 label", "字段 columnName", "字段类型", "字段描述"]
      ]
    }
  ],
  "rewrittenQuestion": "使用指标库改写后的问题"
}
```

**指标库改写示例**：

| 用户问题 | 指标库匹配 | 改写后问题 |
|---------|-----------|-----------|
| "流失客户数有多少" | 流失客户：客户状态包括流失和退款 | "客户状态包括流失和退款的客户数有多少" |
| "同比情况如何" | 同比：目的时间区间和前一年同时间区间 | "今年 X 月和去年 X 月的总数据对比" |
| "环比增长多少" | 环比：目的时间区间和上个月各月数据 | "9 月和 8 月各月的数据对比" |
| "完成情况怎样" | 完成情况：回款金额及预算完成率 | "回款金额及预算完成率是多少" |

**必须包含关联字段**（查询特定项目时）：

| 模型 | 项目关联字段 |
|------|------------|
| 收款登记模型 | `collectionReg3X.project`, `project3X.projectNo` |
| 费用报销 | `project3X.name`, `project3X.projectNo` |
| 项目信息的模型 | `project3X2.projectNo`, `project3X2.projectShortName` |

## 步骤 3：拆分用户问题

**🚨 核心原则**：
1. 大模型直接分析并返回纯 JSON，不调用任何工具
2. 严禁与步骤 2 或步骤 4 合并执行
3. 使用步骤 2 改写后的问题（rewrittenQuestion）作为拆分基础

**输出格式**:
```json
{
  "question": ["问题 1", "问题 2"]
}
```

**依赖关系类型**：

| 类型 | 特征 | 示例 | 执行策略 |
|------|------|------|---------|
| 链式依赖 | 后一个问题需要前一个结果 | `["查项目", "这些项目的收款"]` | 串行 |
| 共同依赖 | 多个问题依赖同一前置问题 | `["查项目", "查收款", "查费用"]` | 先串行后并行 |
| 完全独立 | 问题相互独立 | `["24 年收款", "24 年费用"]` | 并行 |

## 步骤 4：大模型自主执行查询

**🚨 核心原则**：
1. 大模型完全自主规划、判断和决定整个查询流程
2. 严禁与步骤 2 或步骤 3 合并执行
3. 有依赖的查询必须串行执行，无依赖的查询可以批量并行执行
4. **严禁在 Python 代码中导入和调用函数**，直接使用命令行脚本

### DSL 生成（大模型直接输出）

**输出格式**:
```json
{
    "modelName": "使用的模型名称",
    "type": "table",
    "filters": [
        {
            "field": "筛选的字段名称",
            "op": "eq",
            "values": ["筛选的值"]
        }
    ],
    "dimensions": [
        {
            "field": "维度的字段名称",
            "order": "asc",
            "dateFormat": "YM"
        }
    ],
    "metrics": [
        {
            "field": "指标的字段名称",
            "type": "SUM",
            "order": "desc"
        }
    ],
    "limit": 100
}
```

**🚨 关键规则**：
- ✅ 直接输出纯 JSON，不要使用 Markdown 代码块
- ✅ 字段名必须带表前缀（如 `collectionReg3X.receivingAmount`）
- ✅ 必须直接复制步骤 2 的完整 `columnName`
- ✅ `dimensions` 和 `metrics` 至少一个不为空

**DSL 生成要点**：

| 要点 | 说明 |
|------|------|
| 意图判断 | `table`（查明细）vs `chart`（聚合分析） |
| 时间处理 | 年度按自然年处理（如 2024-01-01 到 2024-12-31） |
| 趋势处理 | 一年/一季度按月 (`YM`)，一月按日 (`YMD`) |
| 字段限制 | dimensions + metrics 总计不超过 10 个，但至少有 1 个 |
| 自动排序 | 时间维度或最大/最小指标需加 `order` |
| 百分比转换 | 90% 转换为 0.9 |

### 📅 日期类型过滤格式（重要）

**DSL.md 第 19 行和第 44 行规定的日期过滤格式**：

```json
{
  "field": "collectionReg3X.receivingDate",
  "op": "bte",
  "values": ["2024-01-01 00:00:00", "2024-12-31 23:59:59"]
}
```

| 要素 | 正确写法 | 错误写法 |
|------|---------|---------|
| **操作符** | `"op": "bte"` | `"op": "between"` / `"op": "ge"` + `"op": "le"` |
| **时间格式** | `"2024-01-01 00:00:00"` | `"2024-01-01"` |
| **values 结构** | `["开始时间", "结束时间"]` | 单个值 / 分开两个 filter |

**常见时间范围示例**：

| 时间范围 | values 写法 |
|---------|------------|
| 2024 年全年 | `["2024-01-01 00:00:00", "2024-12-31 23:59:59"]` |
| 2024 年 Q1 | `["2024-01-01 00:00:00", "2024-03-31 23:59:59"]` |
| 2024 年 3 月 | `["2024-03-01 00:00:00", "2024-03-31 23:59:59"]` |
| 去年同期 | `["2023-01-01 00:00:00", "2023-12-31 23:59:59"]` |

## 步骤 5：结果汇总分析

**🚨 核心原则**：
1. **分析总结由大模型直接处理**，不要使用 Bash 工具写长段代码
2. **数据运算必须调用计算器**，不能自己计算
3. **不涉及计算时严禁调用 Bash 工具**，直接输出 Markdown 报告
4. **严禁与步骤 4 合并执行**

### 计算器调用方式

**🚨 核心原则**：
1. 计算器只用于数据运算（加减乘除、百分比、增长率、同比、环比等）
2. 分析总结由大模型直接处理，不要写 Bash 脚本
3. 直接从查询结果提取原始数据让计算器计算
4. 计算器结果必须验证，发现异常时重新拆分计算或人工核验
5. **🚨 禁止心算中间结果**：计算器表达式中必须使用原始数据，不能先心算/笔算出中间结果再填入

**调用示例**：
```bash
# ✅ 正确做法：直接使用原始数据
python3 scripts/calculator.py '[["支出合同总额", "172311.74 + 16743.84 + 16743.84"], ["收支差额", "4005000 - (172311.74 + 16743.84 + 16743.84)"]]'

# ❌ 错误做法：先心算出 205799.42 再填入
python3 scripts/calculator.py '[["支出合同总额", "172311.74 + 16743.84 + 16743.84"], ["收支差额", "4005000 - 205799.42"]]'
```

**计算器结果验证方法**：
1. **数量级检查**：结果是否与原始数据量级匹配
2. **拆分重算**：将复杂计算拆分为多个简单计算分别验证
3. **逻辑检查**：同比/环比结果是否符合业务常识

## 🔄 多轮对话优化

### 上下文复用清单

| 上下文内容 | 来源 | 复用方式 |
|------------|------|----------|
| 会话 ID | 步骤 1 输出 | 直接用于 logger.load_session() |
| 模型列表 | 步骤 2 输出 | 直接复制用于步骤 4 |
| 查询结果 | 步骤 4 输出 | 作为 last_result 传递 |

### 常见场景

| 场景 | 正确做法 |
|------|---------|
| 追问同一个项目 | 复用步骤 2 模型，直接执行步骤 4 |
| 细化查询条件 | 复用模型，调整查询条件执行步骤 4 |
| 切换分析维度 | 复用会话 ID，添加新模型执行步骤 4 |
| 对比分析 | 复用已有结果，直接步骤 5 分析 |
| **跨轮次结果复用** | **直接从上一轮获取答案，不再重复查询** |
| **追问业务详情** | 关联当前话题对象（如项目），默认查该对象的详情，而非全量数据 |

### 上下文关联执行规范

**🚨 多轮对话必须遵守以下规则**：

1. **代词指代识别**：当用户使用"这个"、"该项目"、"他"、"它"等指代词时，必须回溯最近一轮对话中明确提到的具体对象

2. **话题范围锁定**：在同一话题下，追问某业务（收款、付款、合同等）的详情时，**默认范围是当前话题对象**，而非全量数据

3. **明确查询边界**：执行查询前，在内心明确过滤条件（尤其是项目名称、时间等），避免查全量

4. **主动关联上下文**：不等待用户明确说明，主动基于上下文推理查询范围

**上下文关联决策树**：

```
用户问：收款详情
    ↓
上一轮话题是什么？
    ↓
├── 明确提到"某项目" → 直接查该项目收款
├── 是全量查询话题 → 按用户意图查全量或问清范围
└── 无法判断 → 使用"该项目"/"某项目"确认
```

### 跨轮次结果复用规则

> **核心原则**：如果后一轮的问题已经在上一轮得到答案，**大模型直接从对话上下文获取结果，不再重复执行查询**。

**错误做法**：写 Python/Bash 代码读取 `logs/{session_id}/step4-*.json` 文件

**正确做法**：大模型直接从对话历史中找到结果

## ⚠️ 常见错误与解决

| 问题 | 原因 | 解决方法 |
|------|------|---------|
| 合并步骤或跳过步骤 | 急于执行查询 | 每一步单独输出 JSON，按顺序执行 |
| 步骤 2/3 后等待确认 | 误解执行连续性 | 输出 JSON 后立即继续下一步 |
| 步骤 3/5 调用 Bash | 违反禁用工具规定 | 步骤 3 直接输出 JSON；步骤 5 仅计算器可调用 Bash |
| 查询结果包含所有数据 | 未传递上下文 | 后续查询传入 `last_question` 和 `last_result` |
| 步骤 4 盲目执行 | 未分析依赖关系 | 先分析依赖，说明策略，再执行 |
| MapperException/字段名格式错误 | 字段名缺少表前缀 | 直接复制步骤 1 的 `columnName`（如 `collectionReg3X.receivingAmount`） |
| DSL 格式错误 | 聚合类型/字段为空 | `dimensions` 和 `metrics` 至少一个不为空 |
| 计算结果错误 | 大模型心算或笔算 | 直接使用计算器计算原始数据，表达式中不能出现心算的中间结果 |
| 未使用指标库改写问题 | 忽略指标库 | 步骤 2/3 必须使用指标库改写 |
| 同比/环比/时间范围错误 | 对比时间段不一致 | 同环比必须对比相同时间段 |
| 盲目下钻查询 | 没有假设驱动 | 先列出假设清单，再逐个验证 |
| 过滤条件失效 | API 忽略无效枚举值 | 查询前先用分组统计确认枚举值 |
| **枚举值臆想** | **使用臆想的枚举值导致过滤失效** | **先用 GROUP BY 查询确认实际枚举值，再过滤** |

### 案例：过滤条件失效未验证

**问题**：使用"完工已结"查询完工项目，返回 52 条数据（实际应为 6 条）。

4. ❌ 直接报告错误结果

1. ✅ 查询前先用分组统计确认枚举值：`{"type":"chart","dimensions":[{"field":"project3X.projectState"}],"metrics":[{"field":"project3X.name","type":"COUNT"}]}`
2. ✅ 确认实际枚举值为："在建"、"完工"、"保修"、"空"
3. ✅ 使用正确的枚举值"完工"进行过滤查询
4. ✅ 验证过滤结果：查询后检查返回数据的状态字段是否匹配

### 案例：枚举值臆想导致查询错误

**问题**：查询"完工项目数量"时，臆想枚举值为"完工已结"，导致返回错误结果。

**错误过程**：
```
用户：查询一下现在完工项目有多少个

AI：使用过滤条件 {"field":"project3X.projectState","op":"eq","values":["完工已结"]}
结果：返回 52 个项目 ❌（实际是全部数据）
```

**错误原因**：
- API 收到无效的枚举值"完工已结"时，**不会报错**，而是**忽略该过滤条件**
- 导致返回全部 52 个项目，而非真正的完工项目（6 个）
- 未对返回结果进行合理性验证

**正确流程**：
```
步骤 1：先查询枚举值
{"modelName":"项目基础信息_智能问数","type":"chart","dimensions":[{"field":"project3X.projectState"}],"metrics":[{"field":"project3X.name","type":"COUNT"}]}

步骤 2：确认实际枚举值为"完工"

步骤 3：使用正确枚举值查询
{"modelName":"项目基础信息_智能问数","type":"chart","filters":[{"field":"project3X.projectState","op":"eq","values":["完工"]}],"metrics":[{"field":"project3X.name","type":"COUNT"}]}

结果：返回 6 个项目 ✅
```

**🚨 关键规则**：
| 场景 | 正确做法 | 错误做法 |
|------|---------|---------|
| 状态/类型等枚举字段 | 先 GROUP BY 查询确认枚举值 | 直接使用臆想的值 |
| 不确定的字段值 | 先用 table 类型查询明细确认 | 凭空猜测 |
| API 返回数据量异常 | 检查是否过滤条件被忽略 | 直接相信结果 |

**教训**：使用枚举字段过滤前，**必须先用分组查询确认实际枚举值**，严禁臆想！

### 案例：臆想 DSL 操作符导致查询失败

**问题**：使用不存在的 `contains` 操作符进行模糊查询。

**错误过程**：
```
用户：帮我查一下合同名称包含"钢筋"的合同

AI：使用过滤条件 {"field":"expContract3X.name","op":"contains","values":["钢筋"]}
结果：API 返回空数据 ❌
```

**错误原因**：
- DSL.md 第 42 行明确规定操作符只能是 `eq`, `bte`, `gt`, `gte`, `lt`, `lte`, `in`, `ne`
- **`contains` 这个操作符不存在**，API 会忽略该过滤条件
- 未仔细阅读 DSL.md 的规定

**正确做法**：
- 精确匹配：使用 `eq` 操作符 `{"field":"expContract3X.name","op":"eq","values":["钢筋采购合同"]}`
- 模糊匹配：该系统不支持，必须查询所有数据后人工筛选

**🚨 关键规则**：

| 场景 | 正确做法 | 错误做法 |
|------|---------|---------|
| 精确匹配 | `{"op":"eq","values":["钢筋采购合同"]}` | 臆想其他操作符 |
| 模糊匹配 | 查询所有数据，人工筛选 | 臆想 contains/like 等操作符 |
| 范围查询 | `{"op":"bte","values":["2024-01-01","2024-12-31"]}` | 臆想 between/ge+le |

**教训**：DSL 操作符**必须严格按照 DSL.md 规定**，严禁臆想不存在的操作符！
